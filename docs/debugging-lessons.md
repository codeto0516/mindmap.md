# デバッグ教訓集

このドキュメントは、実際のバグ修正経験から得られた重要な教訓をまとめたものです。

## ノードリサイズ時の表示問題（2024年修正）

### 問題の概要
ノードの幅をリサイズすると、他のノードが消えるように見える問題が発生しました。

### 最終的な解決策
React Flowのノードに`height`と`width`を明示的に設定することで解決しました。

```typescript
const nodes = useMemo(
  () =>
    layoutNodes.map((node) => ({
      ...enrichNodeData(node, editingNodeId === node.id),
      selected: selectedNodeIds.includes(node.id),
      // React Flowがノードのサイズを正しく認識するために明示的に設定
      height: nodeHeights[node.id] ?? 36,
      width: nodeWidths[node.id] ?? DEFAULTS.NODE_WIDTH,
    })),
  [layoutNodes, selectedNodeIds, editingNodeId, enrichNodeData, nodeHeights, nodeWidths],
);
```

## 学んだ教訓

### 1. 実際の動作確認が最も重要

**問題点:**
- E2Eテストは通過していたが、実際のブラウザでは問題が発生していた
- テスト環境と実際の環境の違いを見逃していた

**教訓:**
- テストだけでなく、**実際のブラウザでの動作確認**を必ず行う
- ユーザーが報告した「検証ツールでの確認結果」が決定的な手がかりになった
- デバッグ情報（コンソールログ、検証ツール、スクリーンショットなど）を積極的に活用する

### 2. ライブラリの仕様を理解する

**問題点:**
- React Flowはノードのサイズを自動計測するが、リサイズ時には明示的な設定が必要だった
- ライブラリの仕様を十分に理解していなかった

**教訓:**
- 使用しているライブラリのドキュメントや仕様を確認する
- 自動計測される機能でも、特定の条件下では明示的な設定が必要な場合がある
- ライブラリの想定外の挙動がないか、実際の動作で検証する

### 3. 最小限の変更で解決する

**問題点:**
- 最初は複雑な解決策（位置調整ロジック、`fitView`の変更など）を試した
- 最終的には**1行の変更**で解決した

**教訓:**
- 複雑な解決策を試す前に、**シンプルな解決策**を検討する
- 不要な変更を削除し、最小限の修正に絞ることで、コードの保守性が向上する
- 「もっとシンプルに改善して」というユーザーのフィードバックが正しかった

### 4. 仮説を立てて検証する

**検証プロセス:**
1. `fitView`が原因か？ → 検証 → 違った
2. `onNodesChange`の依存関係が原因か？ → 検証 → 違った
3. 高さの再計算が原因か？ → 検証 → 違った
4. 最終的にサイズの明示的設定で解決

**教訓:**
- 仮説を立てて、一つずつ検証していくプロセスが重要
- ただし、早い段階で実際の動作確認を入れると効率的
- 各仮説を検証し、次の仮説に進む

### 5. デバッグ情報の価値

**決定的な情報:**
- ユーザーが提供した「検証ツールで確認したらDOMは消えていなかった。適当にheightを指定したら表示された」

**教訓:**
- デバッグ情報（コンソールログ、検証ツール、スクリーンショットなど）を積極的に活用する
- ユーザーからの具体的な情報が解決の鍵になることが多い
- 問題の本質を見極めるために、様々な角度から情報を収集する

### 6. 問題の本質を見極める

**表面的な問題:**
- 「ノードが消える」ように見えた

**実際の問題:**
- DOMは存在するが表示されない
- React Flowがノードのサイズを正しく認識していない

**教訓:**
- 表面的な症状だけでなく、**問題の本質**を見極めることが重要
- 「消える」という表現から、DOMが削除されていると誤解していた
- 実際には表示の問題だった

### 7. 段階的なアプローチ

**問題点:**
- 一度に複数の変更を加えてしまった

**教訓:**
- 一度に複数の変更を加えず、**1つずつ検証**する
- 不要な変更を削除し、最小限の修正に絞る
- バグ修正後は、本当に必要な変更だけを残す

## デバッグのベストプラクティス

### 1. 問題の再現
- 実際のブラウザで問題を再現する
- 再現条件を明確にする（どのノードを、どのように操作したか）

### 2. 情報収集
- 検証ツールでDOMの状態を確認
- コンソールログでエラーや警告を確認
- スクリーンショットで視覚的な状態を記録

### 3. 仮説の検証
- 複数の仮説を立てる
- 一つずつ検証する
- 検証結果を記録する

### 4. 最小限の修正
- シンプルな解決策を優先する
- 不要な変更を削除する
- コードの保守性を考慮する

### 5. テストと確認
- 修正後、実際のブラウザで動作確認
- E2Eテストだけでなく、手動テストも行う
- ユーザーに確認してもらう

## まとめ

今回のバグ修正から学んだ最も重要な教訓は：

1. **実際の動作確認を最優先にする**
2. **ライブラリの仕様を理解する**
3. **最小限の変更で解決する**
4. **デバッグ情報を活用する**
5. **問題の本質を見極める**

これらの教訓は、今後のバグ修正にも役立ちます。
